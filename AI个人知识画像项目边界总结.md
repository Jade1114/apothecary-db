# AI 个人知识画像项目边界总结

## 一、项目定位

### 项目名称
AI个人知识画像助手

### 项目定义
一个基于用户上传资料的 AI 个人知识画像原型，围绕资料理解、检索增强与结构化画像生成场景进行开发。

### 核心目标
基于用户上传的文档、观点、表达内容，生成结构化的个人知识画像。

换句话说，这个项目当前阶段只做一件事：

- 让 AI 基于“属于这个人的材料”，总结出这个人的特征、偏好、兴趣和表达风格。

---

## 二、当前阶段边界

### 当前阶段要做到的内容

#### 1. 输入
支持用户输入以下内容之一：
- 文本文档
- Markdown 文档
- 个人观点 / 自述
- 聊天记录片段
- 项目说明 / 学习笔记

#### 2. 处理
系统能够完成：
- 读取输入内容
- 对内容进行基础切分
- 做基础检索
- 基于检索结果组织上下文
- 调用大模型生成个人画像

#### 3. 输出
输出一个结构化的个人画像，例如：
- 技术兴趣
- 常关注的话题
- 表达风格
- 思维倾向
- 价值偏好
- 学习方向
- 可归纳的角色标签

当前阶段的重点是：
- 输出有结构
- 输出可读
- 输出尽量有依据

---

## 三、当前阶段明确不做的内容

为了控制项目范围，当前阶段暂不考虑：

- 多用户系统
- 账号权限
- 长期记忆系统
- 自动更新画像
- 复杂 Agent 工作流
- 主动推荐系统
- 复杂知识库管理后台
- 多模态输入
- 本地模型部署
- 复杂 RAG 优化（重排序、多路召回、混合检索等）

一句话概括：

当前阶段只做“资料输入 -> 基础检索 -> 画像生成”这一条主链路。

---

## 四、写进简历的最低完成标准

这个项目达到以下条件后，就可以写进简历：

- 能上传或输入资料
- 能生成结构化个人画像
- 画像不是纯瞎总结，而是基于资料内容
- 最好能展示画像依据片段
- 能清楚讲明白项目链路

---

## 五、这个项目要训练的技术方向

### 1. Prompt Engineering
训练内容：
- 任务说明设计
- 输出结构约束
- 降低模型跑偏
- 提升画像输出稳定性

### 2. 模型 API 调用
训练内容：
- 调用大模型 API
- 组织输入输出
- 请求参数处理
- 基础异常处理

### 3. RAG 基础链路
训练内容：
- 文档输入
- 文本切分
- 检索
- 将检索结果作为上下文输入模型
- 基于资料生成结果

### 4. 结构化输出
训练内容：
- 让模型输出固定结构
- 让结果可读、可解析、可展示
- 将画像从一段散文变成可复用的数据对象

### 5. 基础前后端交互 / AI 原型实现
训练内容：
- 输入提交
- 结果返回
- 页面展示
- 错误反馈

### 6. 基础工程意识
训练内容：
- 项目边界控制
- 文档说明
- 调试
- 结果验证
- 版本迭代意识

---

## 六、推荐技术方向

### 推荐技术路线

#### 后端 / 逻辑层
优先推荐使用：
- Python

原因：
- AI 生态更顺手
- RAG 链路实现成本更低
- 更适合快速做出原型
- 能顺手补足 Python 在 AI 应用开发中的使用能力

#### 前端
可选：
- 简单页面 / 轻前端
- Vue

原因：
- 当前重点不是做复杂前端，而是把 AI 应用链路跑通
- 如果需要更像产品，Vue 是自然选择

#### 模型
- 优先使用现成云模型 API
- 当前阶段不建议折腾本地模型部署

#### 检索 / 存储
- 选择轻量方案即可
- 当前目标是跑通链路，不是堆复杂基础设施

---

## 七、这个项目要训练的核心思想

### 1. 先检索，再生成
不要让模型直接瞎总结，而是先找依据，再输出结果。

### 2. 让 AI 输出有依据
项目重点不是“让 AI 会总结”，而是“让 AI 基于资料总结”。

### 3. AI 项目不是功能越多越好，而是链路越清楚越好
重点在于理清：
- 输入是什么
- 处理中发生了什么
- 输出为什么可信
- 每一步的边界在哪里

### 4. 用结构化结果代替模糊结果
目标不是让模型随便说一段话，而是输出可理解、可展示、可复用的结构化画像。

### 5. AI 应用的价值在于场景，不在于堆名词
当前项目重点是“个人知识画像”这个明确场景，而不是堆砌 Agent、长记忆、本地模型、多模态等高级概念。

---

## 八、当前阶段一句话总结

这个项目第一阶段只做“上传资料后，基于检索增强生成结构化个人画像”，重点训练 Prompt、RAG、模型调用、结构化输出和 AI 应用原型开发能力。

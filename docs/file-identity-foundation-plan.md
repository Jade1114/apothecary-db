# 文件身份层基础重构计划

## 1. 这条分支要做什么

本分支只做一件事：

> 把当前“能扫描 Vault 并导入文件”的基础链路，推进到“能正式识别并管理文件身份”的阶段。

当前系统已经具备：

- 手动触发 Vault 扫描
- 基于文件类型进行 parser 解析
- 生成并落盘标准化文档 `yaml + md`
- 基于标准化文档进入现有 ingest 主链路
- 写入 SQLite 和 sqlite-vec

但当前仍缺少一个关键层：

- 系统还没有正式把 Vault 中的文件当成“可追踪对象”来管理

也就是说，当前系统能导入文件，但还没有真正回答下面这些问题：

- 这个文件是谁
- 这个文件是不是之前已经导入过
- 这个文件是否变更了
- 这个文件当前状态是什么
- 这个文件与 documents 的关系是什么

本分支的目标就是补上这一层。

---

## 2. 这条分支明确不做什么

为了保持边界干净，本分支不做以下内容：

- 不做目录监听器
- 不做自动同步任务系统
- 不做 `sync_jobs` 完整落地
- 不做 `document_blocks` 精细化扩展
- 不做图片 / 视频 / 音频导入
- 不做前端完整产品化改造
- 不做 Electron 接入
- 不做 `profiles` 扩展
- 不做完整 schema 全量重构

也就是说，这条分支只解决：

- 文件身份记录
- 文件与 documents 的正式关联
- 文件状态与最小去重/更新判断

---

## 3. 这条分支的目标结果

本分支完成后，系统应具备下面这些最小能力：

1. 扫描 Vault 时先识别并登记文件身份
2. `files` 表成为正式使用的数据层，而不是只停留在设计文档中
3. 文件记录至少包含路径、扩展名、类型、大小、hash、状态等基础信息
4. `documents` 能正式关联到 `file_id`
5. 重复扫描同一个未变化文件时，系统至少具备最小识别能力
6. 文件变化后，系统能够识别“该文件已变化”这一事实，为后续重建索引打基础

换句话说，这一阶段的目标不是自动同步，而是：

> 先让系统真正认识“文件”本身。

---

## 4. 本分支建议修改的内容

### 4.1 正式启用 `files` 表

本分支需要把 `files` 表从“设计中的表”变成“实际工作中的表”。

第一阶段至少使用这些字段：

- `id`
- `path`
- `name`
- `extension`
- `kind`
- `size`
- `hash`
- `status`
- `created_at`
- `updated_at`
- `last_seen_at`

当前不要求一次把所有高级字段和状态机制都做满，但最小文件身份层必须可用。

### 4.2 新增文件身份服务

建议新增一个围绕 `files` 表的 service，例如：

- `FileIndexService`
- 或 `VaultFilesService`

它负责：

- 根据路径查询文件记录
- 创建新文件记录
- 更新已有文件记录
- 识别文件是否变更
- 返回当前文件对应的内部 `fileId`

这层不要和 parser、ingest 混在一起。

### 4.3 在 Vault 扫描链路中接入文件身份层

当前扫描流程是：

- 找到文件
- 直接 parser
- 直接 ingest

本分支应调整为：

- 找到文件
- 先登记 / 更新 `files`
- 再决定是否需要继续 parser 和 ingest

也就是说，文件身份判断应先于 parser。

### 4.4 `documents` 正式关联 `file_id`

当前 `documents` 虽然已经预留了 `file_id` 字段，但尚未真正成为稳定主关系。

本分支需要让：

- `documents.file_id -> files.id`

真正跑起来。

这意味着后续一篇 document 不再只是“某次导入结果”，而是明确来自某个文件对象。

### 4.5 增加最小状态规则

建议本阶段至少引入下面这些状态语义：

- `active`：当前文件正常存在且可用
- `changed`：文件内容变化，需要重新处理
- `deleted`：文件已不存在
- `error`：最近一次处理失败

不要求把状态机做得很复杂，但要让系统开始具备“文件状态”这个概念。

### 4.6 增加最小测试

本分支至少要补以下测试：

- 首次扫描文件时能够写入 `files`
- 再次扫描同一未变化文件时能够识别重复
- 文件变化后能够识别 hash 变化
- `documents` 能正确带上 `file_id`

目标不是一次测试所有同步边界，而是确保文件身份层站得住。

---

## 5. 本分支推荐实现顺序

建议按下面顺序推进，不要并行发散。

### 第 1 步：明确 `files` 表字段与最小状态语义

先把这一层的边界钉死。

### 第 2 步：实现文件身份 service

把 `files` 的查、建、更新、变化判断收进一个独立 service。

### 第 3 步：接入 Vault 扫描流程

让扫描时先经过文件身份层，再进入 parser / ingest。

### 第 4 步：让 `documents` 正式写入 `file_id`

把文件和 document 的关系接起来。

### 第 5 步：增加最小测试

确保文件身份层和扫描链路有基础验证。

---

## 6. 完成标准

当满足下面这些条件时，这条分支可以认为完成：

- `files` 表被正式使用
- Vault 扫描时会创建或更新文件记录
- 文件记录能够保留最小身份信息与状态
- `documents` 能与 `files` 通过 `file_id` 关联
- 同一文件重复扫描时，系统具有基础识别能力
- 文件变化时，系统能识别变化事实
- 相关最小测试通过

---

## 7. 这条分支完成后，下一步再做什么

这条分支完成后，后续才适合继续推进：

- 目录监听器
- `sync_jobs`
- 真正的自动同步
- 删除 / 移动 / 重建索引策略
- 前端文件状态视图

也就是说，本分支是：

> 从“能导入文件”推进到“能认识并管理文件”的关键基础阶段。

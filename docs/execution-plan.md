# 重构执行计划

## 1. 文档目的

这份文档说明本项目的重构执行顺序，不讨论空泛目标，只回答四件事：

1. 这一阶段要做什么
2. 不做什么
3. 怎么判断做完了
4. 为什么这个顺序不能乱

当前已知前提：

- 后端是 `NestJS`
- 前端是 `React + Vite`
- `data/app.db` 已独立
- `Qdrant` 一期保留
- 模型层先封装，后续再配置化
- 当前重点是先把后端链路做稳，不急着接 Electron

---

## 2. 总体策略

本次重构按“先底座、再能力、再编排、再前端”的顺序推进。

原因很直接：

- 底层不稳，业务模块会反复返工
- 模型能力没统一，RAG 很容易写散
- 后端编排没稳定，前端再改也只是跟着抖
- Electron 更应该放到闭环稳定之后

建议分 5 个阶段执行，每个阶段都要求有明确验收标准，没过验收不要进入下一阶段。

---

## 3. 阶段一：基础底座收敛

### 3.1 目标

先把后端运行底座稳定下来，让后续模块有统一基础。

### 3.2 范围

本阶段只处理基础设施，包括：

- 明确 `backend` 的模块骨架
- 增加 `health` 模块
- 增加 `config` 模块
- 收敛 `database` 模块职责
- 固定 SQLite 路径指向 `data/app.db`
- 明确环境变量与默认配置

### 3.3 不做什么

- 不接 Electron
- 不做完整 RAG
- 不做模型配置中心
- 不追求复杂前端页面

### 3.4 交付物

- 后端模块目录结构清晰
- 健康检查接口可用
- 配置读取方式统一
- 数据库访问入口统一

### 3.5 验收标准

- `/health` 返回成功状态
- 数据库路径不再散落在多个文件里
- 任何模块访问配置都通过统一入口
- 任何模块访问 SQLite 都通过 `database` 模块

### 3.6 顺序

1. 建立 `config`
2. 建立 `health`
3. 收敛 `database`
4. 清理直接耦合代码

---

## 4. 阶段二：领域模块收边界

### 4.1 目标

把现有业务模块从“先能用”整理成“职责明确”。

### 4.2 范围

聚焦当前已有或接近已有的业务模块：

- `documents`
- `profiles`
- `ingest`

需要完成的事情：

- 明确每个模块只负责什么
- 拆掉混杂在 controller 里的业务逻辑
- 统一 DTO、返回结构、状态字段
- 让导入、文档、画像三个领域对象边界清晰

### 4.3 不做什么

- 不做完整向量检索能力
- 不做完整 LLM 编排
- 不追求一次性把所有历史代码都重写

### 4.4 交付物

- `documents` 管理文档主数据
- `profiles` 管理画像结果
- `ingest` 只负责导入编排
- 模块之间的依赖方向清楚

### 4.5 验收标准

- `documents` 不再承担 embedding / Qdrant 逻辑
- `profiles` 不再承担通用 RAG 流程
- `ingest` 成为明确的导入入口
- controller 主要负责请求和响应，核心逻辑进入 service
- 模块之间没有明显循环依赖

### 4.6 顺序

1. 先收 `documents`
2. 再收 `profiles`
3. 最后整理 `ingest`

---

## 5. 阶段三：AI 能力层封装

### 5.1 目标

把与模型相关的底层能力从业务模块中抽出来，形成可复用的能力层。

### 5.2 范围

新增并稳定以下模块：

- `embedding`
- `vector`
- `llm`

需要完成的事情：

- 抽象 embedding 调用接口
- 抽象 Qdrant 读写与检索接口
- 抽象 LLM 调用接口
- 让上层业务不再直接感知模型实现细节

### 5.3 不做什么

- 不上复杂 provider 管理页面
- 不做多模型热切换 UI
- 不急着支持很多模型供应商

### 5.4 交付物

- `embedding` 可以稳定把文本转向量
- `vector` 可以稳定写入和检索 Qdrant
- `llm` 可以稳定生成文本结果
- 业务模块可以通过统一 service 调用这些能力

### 5.5 验收标准

- `ingest` 不直接写模型 SDK 细节
- `rag` 可用统一接口调用 embedding/vector/llm
- 更换具体模型实现时，上层业务代码改动有限
- Qdrant 的 collection、point、search 操作有统一入口

### 5.6 顺序

1. 先做 `embedding`
2. 再做 `vector`
3. 最后做 `llm`

---

## 6. 阶段四：RAG 主流程闭环

### 6.1 目标

让系统形成“导入 → 检索 → 回答 → 展示”的最小闭环。

### 6.2 范围

本阶段核心是 `rag` 模块落地，包括：

- 问题输入
- 查询向量生成
- Qdrant 召回
- 文档片段补充
- prompt 组装
- LLM 生成答案
- 返回引用依据

同时需要回补 `ingest`，确保导入数据能被 `rag` 正常消费。

### 6.3 不做什么

- 不追求复杂工作流引擎
- 不做高级 rerank、多路召回、多跳推理
- 不做大型提示词管理平台

### 6.4 交付物

- 用户可以发起问答请求
- 后端能完成一次完整 RAG 流程
- 前端能看到答案和引用片段
- 出错时能返回清晰错误信息

### 6.5 验收标准

- 至少一类文档可被成功导入并索引
- 至少一个问答接口可以稳定返回结果
- 返回结果中包含答案正文和来源信息
- 当 Qdrant 或模型不可用时，系统能给出明确错误
- 同一问题重复测试时，链路稳定可复现

### 6.6 顺序

1. 修通 ingest 到向量库
2. 建立 rag 查询入口
3. 拼装上下文与 prompt
4. 接入 llm 输出
5. 补错误处理与日志

---

## 7. 阶段五：前端对齐与桌面化准备

### 7.1 目标

让前端稳定消费后端能力，并为后续 Electron 做准备，但本阶段仍然不正式接入 Electron。

### 7.2 范围

- 前端按业务能力重组页面和 API 调用
- 统一对接 `/documents`、`/profiles`、`/ingest`、`/rag`
- 补充状态展示：加载中、失败、空状态
- 检查本地运行体验是否顺畅
- 为 Electron 保留接口边界和目录空间

### 7.3 不做什么

- 不做 Electron 主进程集成
- 不做桌面安装包
- 不做系统托盘、自动更新等桌面增强项

### 7.4 交付物

- 前端页面能稳定调用后端
- 关键流程有清楚反馈
- API 边界稳定
- 后续若要接 Electron，不需要大改后端

### 7.5 验收标准

- 前端可以完成导入、查看文档、发起问答、查看画像
- 所有请求都通过统一 API 层访问后端
- 页面出现失败时能展示清楚的错误提示
- 前后端职责边界明确，没有前端绕过后端直接做数据层操作

### 7.6 顺序

1. 先整理 API 调用层
2. 再整理页面结构
3. 最后补状态与错误反馈

---

## 8. 阶段之间的依赖关系

```text
阶段一 基础底座
→ 阶段二 领域模块收边界
→ 阶段三 AI 能力层封装
→ 阶段四 RAG 主流程闭环
→ 阶段五 前端对齐与桌面化准备
```

原因如下：

- 没有底座，后面不稳
- 没有领域边界，AI 能力会乱挂
- 没有 AI 能力层，RAG 会写死
- 没有 RAG 闭环，前端很难对齐
- 没有稳定前后端边界，Electron 只会增加复杂度

---

## 9. 每阶段完成后的检查清单

### 9.1 架构检查

- 有没有新增跨层直连
- 有没有模块职责继续变混
- 有没有把临时方案写成长期依赖

### 9.2 代码检查

- controller 是否过重
- service 是否职责过大
- 配置是否仍有散落
- 数据访问是否绕过 `database` 模块

### 9.3 功能检查

- 主流程是否还能跑通
- 错误提示是否清楚
- 是否因为重构破坏已有接口

### 9.4 学习检查

每阶段结束后都应能回答：

- 这一层为什么存在
- 这一模块为什么单独拆出
- 如果以后要替换模型，应该改哪一层
- 如果以后接 Electron，哪些层不应该被影响

---

## 10. 最终达标标准

当下面这些条件同时满足时，可以认为这次重构基本达标：

- 后端模块职责明确
- SQLite 与 Qdrant 的职责分离清楚
- embedding / vector / llm 已封装成独立能力
- ingest 与 rag 形成可运行闭环
- 前端只做展示与交互，不承担核心编排
- Electron 仍未接入，但已具备低风险接入前提

---

## 11. 一句话结论

> 先稳后端底座，再收业务边界，再抽 AI 能力，再做 RAG 闭环，最后才考虑 Electron 桌面壳。

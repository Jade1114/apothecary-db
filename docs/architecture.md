# 桌面软件整体架构说明

## 1. 文档目的

这份文档说明当前项目在重构阶段的整体软件架构，重点回答三个问题：

1. 前端、后端、数据层、Qdrant、模型层分别负责什么。
2. 为什么当前阶段先把后端和数据链路做稳，再考虑接入 Electron。
3. 后续扩展时，哪些地方应该稳定，哪些地方允许替换。

这是一份面向项目驱动学习的架构说明，目标不是追求“大而全”，而是帮助你先把系统拆清楚、边界定清楚、执行顺序排清楚。

---

## 2. 当前仓库状态

当前仓库已经具备以下基础：

- `backend` 使用 `NestJS`
- `frontend` 使用 `React + Vite`
- SQLite 数据库文件已经独立到 `data/app.db`
- 一期保留 `Qdrant`，用于向量检索能力
- 模型层暂时先按代码接入，后续再做“配置化切换”

这意味着：项目已经从“单体原型”开始往“可维护的分层应用”演进，但还没有进入“桌面壳集成优化”阶段。

---

## 3. 整体目标

这个项目的核心目标，不是先做一个“看起来像桌面软件”的壳，而是先把下面这条业务链路跑通并稳定下来：

1. 文档进入系统
2. 文档被清洗、切块、建索引
3. 结构化信息进入本地数据库
4. 向量进入 Qdrant
5. 查询请求经过检索与召回
6. 大模型基于检索结果生成回答
7. 前端把结果稳定展示出来

先把这条链路做稳，后面无论是 Web 形态还是 Electron 桌面形态，都只是“承载方式”不同，不会影响核心能力。

---

## 4. 总体架构分层

建议采用下面这套分层思路：

```text
┌──────────────────────────────┐
│ Frontend (React + Vite)      │
│ 界面展示、用户交互、状态管理  │
└──────────────┬───────────────┘
               │ HTTP / JSON
┌──────────────▼───────────────┐
│ Backend (NestJS)             │
│ 业务编排、接口暴露、流程控制  │
└──────┬──────────┬────────────┘
       │          │
       │          │
┌──────▼──────┐  ┌▼────────────────┐
│ Data层      │  │ 模型能力层       │
│ SQLite      │  │ Embedding / LLM │
│ 结构化持久化 │  │ 外部模型封装     │
└──────┬──────┘  └──────┬──────────┘
       │                │
       │                │
┌──────▼────────────────▼──────┐
│ Qdrant                        │
│ 向量存储、相似度检索、召回     │
└──────────────────────────────┘
```

这套分层的关键点只有一句话：

- 前端负责“交互”
- 后端负责“编排”
- 数据层负责“存储”
- Qdrant 负责“向量检索”
- 模型层负责“模型调用”
- 不要把这些职责混在一起

---

## 5. 各层职责划分

### 5.1 Frontend：界面层

`frontend` 的职责应该尽量单纯：

- 展示文档列表、画像、检索结果、问答结果
- 发起用户操作，例如上传、导入、检索、问答
- 维护页面状态和交互反馈
- 不直接处理数据库
- 不直接调用 Qdrant
- 不直接拼装复杂业务流程

前端应该把自己当成“后端能力的使用者”，而不是“业务逻辑中心”。

建议目录结构：

```text
frontend/src
├─ app
├─ pages
├─ features
├─ components
├─ services
├─ hooks
├─ types
└─ utils
```

### 5.2 Backend：应用核心层

`backend` 是当前阶段最重要的一层，它承担系统的“业务中台”职责：

- 对外提供稳定 API
- 承接前端请求
- 调用数据库、Qdrant、模型服务
- 组织导入、切块、嵌入、检索、回答等流程
- 统一处理配置、日志、错误、健康检查

可以把它理解成：这个项目的核心能力都应该先在后端稳定，再交给前端消费。

### 5.3 Data：结构化数据层

当前 `data/app.db` 已经独立出来，这是正确方向。

数据层主要负责：

- 保存文档主记录
- 保存分段、任务、画像等结构化数据
- 保存本地业务状态
- 为后端提供稳定可查询的数据基础

SQLite 在当前阶段是够用的，因为：

- 本地桌面场景天然适合单机数据库
- 零部署成本，学习和调试门槛低
- 对原型和一期重构非常友好

要注意的是，SQLite 负责的是“结构化事实”，不是“语义检索”。

### 5.4 Qdrant：向量检索层

Qdrant 一期保留，定位非常明确：

- 存储文本分段的向量
- 根据用户查询做相似度检索
- 返回候选片段给 RAG 流程使用

Qdrant 不负责：

- 文档主数据管理
- 用户配置管理
- 最终答案生成

也就是说，Qdrant 是“检索引擎”，不是“业务数据库”。

### 5.5 模型层：可替换能力层

模型层包含两类能力：

1. Embedding：把文本转成向量
2. LLM：基于上下文生成结果

当前阶段先按代码封装，后续再做配置化。这样安排是务实的，原因有三个：

- 现在最重要的是先把业务链路打通
- 如果一开始就做复杂配置系统，很容易把精力耗在框架上
- 模型层天然适合后续抽象成 provider 机制，不必一开始过度设计

建议模型层对上只暴露统一接口，对下再分别接不同供应商或本地模型。

## 6. 推荐的数据流

### 6.1 文档导入流

```text
前端上传/触发导入
→ 后端 ingest 流程启动
→ 文档清洗与切块
→ 结构化元数据写入 SQLite
→ 分段向量写入 Qdrant
→ 返回导入结果给前端
```

### 6.2 检索问答流

```text
前端提交问题
→ 后端生成 query embedding
→ 到 Qdrant 检索相关片段
→ 从 SQLite 补全文档元信息
→ 后端组装 prompt
→ 调用 LLM 生成答案
→ 返回答案 + 引用片段给前端
```

### 6.3 画像生成流

```text
前端请求生成画像
→ 后端读取相关文档
→ 后端做聚合、抽取或总结
→ 写回 SQLite 中的 profiles 数据
→ 前端读取并展示
```

---

## 7. 为什么现在先不接 Electron

### 7.1 先接 Electron，收益不大

Electron 解决的是“桌面壳”和“本地能力打包”的问题，比如：

- 应用窗口
- 本地菜单
- 文件系统桥接
- 应用分发

但当前项目真正不稳定的地方并不在这里，而在后端业务链路：

- 数据模型还在收敛
- 模块职责还没完全定型
- 导入、切块、嵌入、检索、回答链路还需要打磨
- 模型接入方式还没完成抽象

如果现在就把 Electron 接进来，等于在核心能力还不稳的时候，又多引入一层桌面壳复杂度。

### 7.2 Electron 会放大调试复杂度

一旦接入 Electron，开发时要同时关注：

- 前端页面问题
- 后端服务问题
- Electron 主进程问题
- Renderer 与主进程通信问题
- 打包与路径问题
- 本地数据库与资源目录定位问题

对零基础、按项目驱动学习的人来说，这会明显增加排障难度。

### 7.3 先稳后端，后续接壳成本更低

如果后端 API、数据模型、检索流程都已经稳定，那么 Electron 只是增加一个桌面入口，工作会清晰很多：

- 前端页面基本不变
- 后端接口基本不变
- 数据层位置基本不变
- Electron 只负责把现有能力封装成桌面应用

一句话概括：

> Electron 不是当前阶段的核心能力，它只是交付形态；后端能力链路才是当前阶段必须先做稳的东西。

---

## 8. 当前阶段的架构原则

- 以后端为中心组织能力
- 结构化数据与向量数据分开存
- 模型层先封装，后配置化
- 模块边界优先于功能堆叠
- 先跑通闭环，再追求桌面体验

## 10. 本阶段结论

当前阶段的正确重构方向是：

1. 以前后端分离为基础
2. 以后端模块化为核心
3. 以 SQLite + Qdrant 为双存储基础
4. 以模型层封装为未来扩展点
5. 先不接 Electron，先把后端链路做稳

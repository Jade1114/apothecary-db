# 本地知识库文件夹模式设计说明

## 1. 文档目的

本文档定义本项目后续面向个人长期使用的核心存储与知识组织模式。

这份设计不是为了迎合通用市场产品形态，而是明确回答一个更重要的问题：

> 如果这个软件最终只服务于我自己，它应该怎样建立在我的本地文件系统之上，成为一个长期可用的个人知识引擎？

本文档重点说明：

- 为什么本项目后续应该采用“本地知识库文件夹”模式
- 原始文件、SQLite、向量库三者分别负责什么
- 文件新增、删除、修改、移动时，系统应该如何同步
- 后续桌面版为什么应以文件系统为根，而不是以软件私有数据库为根

---

## 2. 背景与出发点

当前项目已经完成了最小可工作的 RAG 后端与基础前端工作台，具备：

- 文档入库
- 文本切块
- embedding 调用
- 向量写入 Qdrant
- RAG 检索
- LLM 基于 evidence 回答

但当前这一阶段仍然偏“应用功能验证”，还没有完全进入长期桌面软件形态。

如果软件未来只是给自己用，那么产品设计的优先级就不应该是“更像市场上的通用 AI 工具”，而应该是：

- 是否长期可控
- 是否和本地文件整理习惯兼容
- 是否不会把原始知识锁死在软件内部
- 是否能够随着文件持续积累而长期演进

基于这个目标，本项目后续建议采用：

**“本地知识库文件夹 + 软件索引层 + 向量检索层”**

这是一种更适合个人长期使用的软件形态。

---

## 3. 核心设计目标

本模式的目标不是把软件做成“另一个云端聊天工具”，而是把它做成一个建立在本地文件系统上的知识增强引擎。

核心目标如下：

### 3.1 原始知识归用户所有

软件不应该拥有你的原始知识文件。

原始文件应继续存在于你的本地文件系统中，软件只围绕这些文件建立：

- 索引
- 解析结果
- 向量表示
- 检索能力
- 回答能力

### 3.2 文件系统是知识源头

本项目后续应以一个本地目录作为知识库根目录，类似于 Obsidian Vault，但不局限于 Markdown。

软件应当理解：

- 这个目录里的文件是什么
- 哪些文件有变化
- 哪些内容需要重新解析
- 哪些索引需要新增、更新或删除

### 3.3 软件是“理解层”和“调用层”

这个软件不是单纯的文件浏览器，也不是单纯的数据库前端。

它的定位应该是：

- 文件系统之上的索引层
- 语义检索层
- RAG 问答层
- 个人知识画像层

### 3.4 面向长期个人使用

这个模式应适配长期积累、长期演化，而不是一次性导入、一次性生成结果。

---

## 4. 核心理念

本项目后续应坚持如下原则：

### 4.1 文件系统优先

原始文件以本地文件夹为根，不被软件接管。

### 4.2 索引与知识分离

- 文件本体留在文件系统
- 索引与状态留在 SQLite
- 向量与语义表示留在 Qdrant

### 4.3 自动同步而非手工维护

用户只需要维护文件，软件负责理解和同步。

### 4.4 回答尽量基于本地资料

系统回答应主要基于用户自己的文件内容，而不是泛泛生成。

一句话概括：

> 原始知识属于用户，索引与理解属于软件。

---

## 5. 目标形态：本地知识库文件夹

后续建议引入一个“知识库根目录”概念。

例如：

```text
KnowledgeVault/
  inbox/
  notes/
  backend/
  ai/
  career/
  journal/
  images/
  music/
  videos/
  projects/
  chats/
```

这里面的内容不局限于 Markdown，而可以包括多种类型的文件，例如：

- `md`
- `txt`
- `pdf`
- `docx`
- 聊天记录导出
- `image`
- `movie`
- `music`
- 项目说明文档
- 其他用户长期保留的个人资料

这一层是：

- 用户可见的
- 用户可直接操作的
- 用户可分类整理的
- 用户可备份迁移的

这就是整个系统的**知识源层**。

## 6. 三层存储结构

后续系统建议采用三层结构。

### 6.1 第一层：用户原始资料层

这一层就是本地知识库目录。

例如：

```text
KnowledgeVault/
  notes/
  projects/
  images/
  videos/
  music/
  docs/
```

这一层负责：

- 保存原始文件
- 体现用户自己的分类方式
- 作为软件的扫描与索引输入

这一层不应该被软件重写成私有格式，也不应该要求用户把知识导入到某种不可见的黑盒结构中。

### 6.2 第二层：软件索引层（SQLite）

这一层建议继续使用 `SQLite`。

这一层不存原始文件本体，而存围绕文件建立的结构化索引和业务状态。

可能包括：

- 文件路径
- 文件类型
- 文件 hash
- 文件大小
- 最后修改时间
- 内部 fileId
- documentId
- chunk 数量
- 当前索引状态
- 解析状态
- 错误状态
- 画像结果
- 查询历史
- 文件与向量索引的映射关系

这一层负责回答：

- 这个文件是谁
- 这个文件上次处理到哪里了
- 这个文件有没有变
- 它对应哪些 chunk
- 它是否已经建立向量索引

### 6.3 第三层：语义检索层（Qdrant）

这一层建议继续使用 `Qdrant`。

这一层负责：

- 保存 chunk 向量
- 保存与检索相关的 payload
- 为 RAG 检索提供召回能力

Qdrant 中保存的内容主要是：

- chunk vector
- chunk payload
- documentId
- fileId
- chunkIndex
- 内容片段
- 来源类型
- 来源名称

这一层解决的是：

- 相似度检索
- evidence 召回
- 向量查询

它不是业务数据库，也不是原始知识源。

## 7. 文件变化同步机制

如果采用本地知识库目录模式，那么软件必须具备文件变化同步能力。

后续至少应支持四类变化：

- 新增文件
- 修改文件
- 删除文件
- 移动文件

### 7.1 新增文件

当用户向知识库目录新增文件时：

1. 软件发现新文件
2. 记录文件元数据
3. 如果文件可解析，则提取文本
4. 做 chunk
5. 写入 SQLite
6. 生成 embedding
7. 写入 Qdrant

### 7.2 修改文件

当文件内容被修改时：

1. 软件检测到修改时间 / hash 变化
2. 找到该文件对应的旧索引记录
3. 更新 SQLite 中的元数据
4. 删除或覆盖旧 chunk 索引
5. 重新解析文本、chunk、embedding
6. 重写 Qdrant 中对应向量

### 7.3 删除文件

当文件被删除时：

1. 软件发现该路径已不存在
2. 找到内部 fileId / documentId
3. 删除 SQLite 中关联记录
4. 删除 Qdrant 中对应的向量点
5. 清理该文件关联的画像或缓存状态（按策略）

这正对应你的需求：

> 如果我删掉了一个个人见解文档，软件应定位到与该文档相关的数据，并在 SQLite 和向量数据库中清理。

### 7.4 移动文件

当文件被移动时，系统最好不要简单地当作“删除 + 新增”。

更理想的处理方式是：

1. 通过 hash / 指纹识别它仍然是同一个文件
2. 更新其路径记录
3. 保留已有 fileId、documentId 和索引关系
4. 只在必要时重建元数据，不重复构建全部向量

这会显著提升长期使用的稳定性。

## 8. 文件身份设计

如果要正确处理新增、删除、修改、移动，就不能只靠“文件路径”识别文件。

后续建议引入三个层次的身份：

### 8.1 文件路径（path）

表示文件当前所在位置。

用途：

- 用户可见
- 目录扫描入口
- UI 展示来源位置

### 8.2 文件指纹（hash）

表示文件内容层面的身份。

用途：

- 判断内容是否变化
- 辅助识别文件是否被移动
- 避免重复入库

### 8.3 内部 fileId

表示软件内部的稳定主键。

用途：

- 关联 SQLite 中的业务记录
- 关联 chunk
- 关联向量点
- 关联画像和历史记录

这种设计可以让系统更稳定地回答：

- 这是一个新文件吗？
- 这是旧文件的新位置吗？
- 这是旧文件的新版本吗？
- 哪些索引应该删，哪些应该保留？

## 9. 多类型文件支持策略

你明确希望知识库目录不只包含 Markdown，而是可以包含各种类型的文件。

这是正确方向，但后续实现应采用“分层支持”而不是“一步到位全理解”。

### 9.1 强理解类型

建议优先支持这些类型进入主 RAG 链路：

- `md`
- `txt`
- `pdf`
- `docx`
- 聊天记录导出文本
- 项目说明类文档

这些文件具备明确的文本内容，适合：

- 解析
- chunk
- embedding
- RAG 检索

### 9.2 弱理解类型

这些类型建议先纳管，但不强行深度理解：

- `image`
- `movie`
- `music`

第一阶段可以先支持：

- 文件存在记录
- 文件路径、名字、大小、时间等元数据
- 可选人工备注
- 可选后续摘要入口

也就是说：

- **先统一纳管所有文件类型**
- **但只让文本类文件进入 RAG 主链路**

后面再逐步扩展：

- OCR
- 图片 caption
- 视频字幕提取
- 音频转录

这样不会因为多模态把主线拖死。

## 10. 与 Obsidian 的关系

这个模式和 Obsidian 有相似之处，但定位并不相同。

### 10.1 相似点

- 都以本地文件夹为知识根
- 都强调用户对原始文件的控制权
- 都支持长期积累

### 10.2 不同点

Obsidian 更偏：

- 笔记编辑
- 链接组织
- 人工书写和浏览

本项目后续更偏：

- 自动索引
- 自动同步
- RAG 检索
- 回答生成
- 个人知识画像

可以把两者理解成：

- Obsidian 负责“写和整理”
- 本项目负责“理解和调用”

后续完全可以支持：

- 直接把某个 Obsidian Vault 作为知识源目录

这是天然兼容的。

## 11. 系统中的角色分工

为了避免后续实现混乱，需要明确文件系统模式下各层责任。

### 11.1 文件系统层

负责：

- 保存原始文件
- 反映用户真实分类方式

### 11.2 文件理解层

负责：

- 扫描目录
- 解析文件类型
- 提取文本或元数据
- 判断变化类型

### 11.3 索引层（SQLite）

负责：

- 文件记录
- 文件状态
- chunk 记录
- 索引状态
- 画像与查询历史

### 11.4 向量层（Qdrant）

负责：

- chunk 向量保存
- 检索召回

### 11.5 生成层（LLM）

负责：

- 基于 evidence 回答
- 基于文件资料生成画像和总结

### 11.6 前端 / 桌面层

负责：

- 选择知识库目录
- 查看索引状态
- 查看文件解析结果
- 提问、查看回答和 evidence
- 管理知识库设置

## 12. 推荐的数据流

### 12.1 文件导入 / 同步流

```text
用户在知识库目录中新增/修改文件
→ 软件扫描或监听到文件变化
→ 解析文件内容
→ 更新 SQLite 文件记录
→ 切块并生成 embedding
→ 写入 Qdrant
→ 更新索引状态
```

### 12.2 删除同步流

```text
用户删除文件
→ 软件发现路径失效
→ 找到对应 fileId/documentId
→ 删除 SQLite 中关联记录
→ 删除 Qdrant 中关联向量
→ 更新历史状态
```

### 12.3 问答流

```text
用户提问
→ 软件生成 query embedding
→ Qdrant 检索 evidence
→ LLM 基于 evidence 回答
→ 返回 answer + evidence
```

## 13. 当前架构如何演进到这个模式

当前项目已经具备：

- `backend`: NestJS
- `frontend`: React + Vite
- `data/app.db`: SQLite
- `Qdrant`: 向量库
- 最小 RAG 闭环

因此，后续不需要推翻重来，而是在现有基础上自然扩展。

建议新增或强化的能力包括：

- `vault`：知识库目录管理
- `filesystem`：目录扫描与监听
- `parser`：多类型文件解析
- `sync`：增量同步编排
- `ingest`：从“手动文本入库”演进到“文件驱动入库”

也就是说，当前架构是可以平滑演进到这个模式的。

## 14. 后续功能演进建议

### 14.1 第一阶段

- 用户手动指定知识库目录
- 软件全量扫描目录
- 优先支持文本类文件入库
- 支持基于这些文件做问答

### 14.2 第二阶段

- 监听目录变化
- 自动增量索引
- 文件删除同步清理
- 移动文件识别

### 14.3 第三阶段

- 多目录支持
- 忽略规则（类似 `.gitignore`）
- 文件标签与分类增强
- Obsidian Vault 兼容优化

### 14.4 第四阶段

- 图片 OCR
- 音频转录
- 视频字幕提取
- 多模态资料进入统一知识体系

---

## 15. 当前阶段暂不做的内容

为了避免范围失控，当前阶段不建议优先做：

- 多用户协作
- 云端同步
- 权限系统
- 跨设备实时协同
- 大规模自动工作流编排
- 所有类型文件的一步到位深度理解

当前更重要的是先把：

- 本地文件系统 → 索引 → 检索 → 回答

这条主链做稳。

---

## 16. 结论

如果这个软件的核心目标是“只给自己长期使用”，那么最合理的形态不是把知识锁进软件，而是：

- 以本地文件系统为根
- 以知识库目录为知识源
- 由软件自动扫描、索引和同步
- 用 SQLite 管理结构化状态
- 用 Qdrant 管理语义检索
- 用 RAG 和 LLM 生成回答与画像

也就是说，后续本项目应当朝这个方向演进：

> 一个面向个人长期使用、以本地文件系统为知识源、以自动索引和 RAG 为核心能力的桌面知识引擎。

这不是通用市场意义上的“内容平台”，而是更贴近你自己的使用方式、更适合长期积累的个人知识操作系统。

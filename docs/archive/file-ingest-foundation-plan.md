# 文件导入基础链路重构计划

## 1. 这条分支要做什么

本分支只做一件事：

> 把当前以“手动输入文本”为主的 ingest 入口，推进到“文件导入基础链路”的第一阶段。

这一阶段只覆盖：

- 文本文件与富文本文件识别
- parser 解析
- 标准化文档生成
- 标准化文档落盘为 `yaml + md`
- 基于标准化文档调用现有 ingest 主链路

当前只考虑第一阶段支持的文件类型：

- `txt`
- `md`
- `pdf`
- `docx`

---

## 2. 这条分支明确不做什么

为了保持边界清晰，本分支不做以下内容：

- 不做图片 OCR
- 不做视频 / 音频解析
- 不做 Electron 接入
- 不做前端完整产品化改造
- 不做完整文件系统监听
- 不做更大范围的 SQLite schema 全量重构
- 不做多 provider 向量层设计
- 不重新讨论 sqlite-vec 路线

也就是说，这条分支只解决：

- 文件怎样进入系统
- 文件怎样先变成标准化文档
- 标准化文档怎样进入当前 ingest

---

## 3. 这条分支的目标结果

完成后，项目应具备下面的最小能力：

1. 能接收一个文本/富文本文件路径作为输入
2. 能根据文件类型选对应 parser
3. 能把文件内容转成统一的标准化文档对象
4. 能把标准化文档落盘成 `yaml + md`
5. 能把标准化文档中的正文内容送入当前 ingest 主链路
6. 能最终完成 chunk、embedding、sqlite-vec 索引写入

换句话说，这一阶段的目标不是“做完整本地知识库系统”，而是：

> 先把“文件 -> 标准化文档 -> ingest”这条链路跑通。

---

## 4. 本分支建议修改的内容

### 4.1 新增 `parser` 模块

本分支需要新增一个基础 parser 层，用来处理不同文件类型。

建议先做最小结构，例如：

```text
backend/src/parser/
  parser.module.ts
  parser.service.ts
  types/
  parsers/
    text-file.parser.ts
    markdown-file.parser.ts
    pdf-file.parser.ts
    docx-file.parser.ts
```

第一阶段只需要保证：

- 能按扩展名路由 parser
- 能输出统一结构

不要求一开始把 parser 做成高度复杂体系。

### 4.2 定义 `NormalizedDocument` 类型

需要定义统一的标准化文档类型，作为 parser 输出与 ingest 输入之间的中间层。

这一层应包含：

- 文件来源信息
- 标题
- 正文文本
- 元数据
- 标准化文档落盘路径

这一层是整个后续文件系统知识库模式的基础。

### 4.3 新增标准化文档落盘逻辑

需要新增一个明确的落盘逻辑，把标准化文档保存成：

- `yaml frontmatter`
- `markdown body`

落盘目录可以先做成软件内部固定目录，不在这一阶段讨论最终桌面路径策略。

第一阶段重点是：

- 先能稳定生成并保存标准化文档
- 后续需要重建时可以直接从这层恢复

### 4.4 新增文件导入入口

当前 `ingest` 是基于文本内容直接输入。

本分支需要新增一个“文件导入入口”，它负责：

- 接收文件路径
- 识别类型
- 调 parser
- 生成标准化文档
- 调现有 ingest 管线

这个入口可以作为新的 service / controller 路径存在。

重点不是前端 UI，而是后端先把链路打通。

### 4.5 调整当前 ingest 的职责定位

本分支不要求完全重写 ingest，但需要明确把 ingest 收敛为：

- **处理标准化后的文本内容**

也就是说，ingest 不再负责“理解原始文件”，而只负责：

- chunk
- embedding
- vector index
- 结构化索引写入

### 4.6 增加最小测试

本分支需要补最小测试覆盖以下关键点：

- parser 路由是否正确
- 标准化文档是否成功生成
- 标准化文档是否成功落盘
- 文件导入入口是否能调通 ingest

目标不是追求全量测试，而是确保第一阶段主链路可验证。

---

## 5. 本分支的推荐实现顺序

建议严格按下面顺序做，不要并行发散。

### 第 1 步：定义 `NormalizedDocument`

先把统一中间格式定下来。

### 第 2 步：实现最小 parser 框架

先把：

- `txt`
- `md`
- `pdf`
- `docx`

四种文件的 parser 接口和基础实现搭起来。

### 第 3 步：实现标准化文档落盘

先能把 parser 输出稳定写成 `yaml + md` 文件。

### 第 4 步：实现文件导入 service

打通：

- 文件路径
- parser
- normalized document
- ingest

### 第 5 步：增加最小测试与验证脚本

确保这条链路可以被稳定验证。

---

## 6. 完成标准

当满足下面这些条件时，这条分支可以认为完成：

- 能输入一个本地文本/富文本文件
- 系统能自动识别类型并调用 parser
- 能输出统一的 `NormalizedDocument`
- 能将标准化文档落盘为 `yaml + md`
- 能基于标准化文档正文调通当前 ingest
- 最终能完成 sqlite-vec 索引写入
- 有最小测试覆盖这条新链路

---

## 7. 这条分支完成后，下一步再做什么

这条分支完成后，后续才适合进入下一阶段，例如：

- `files / documents / document_blocks / chunks / chunk_vectors` 的更完整 schema 落地
- 文件系统扫描与同步
- 去掉手动文本入口
- 前端从“文本工作台”转向“文件导入工作台”

也就是说，本分支是：

> 文件系统驱动知识库模式的第一块真正工程化落地基础。
